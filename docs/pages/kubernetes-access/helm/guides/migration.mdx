---
title: Migrating from a legacy version of the siriusec Helm chart
description: How to migrate a Siriusec cluster using the legacy siriusec helm chart to the siriusec-cluster Helm chart
---

In this guide, we'll detail a way to migrate an existing Siriusec cluster using the legacy `siriusec` Helm chart
to use the newer `siriusec-cluster` Helm chart instead.

<Admonition type="warning">
  This guide details a very simple migration scenario for a smaller Siriusec cluster which is not deployed for high availability.

  If your Siriusec cluster is required to support many users and should be deployed in a highly available configuration, you should
  consider [following a different guide](../guides.mdx) and storing your cluster's data in AWS DynamoDB or Google Cloud Firestore.
</Admonition>

(!docs/pages/kubernetes-access/helm/includes/helm-install.mdx!)

## Step 3. Get the Siriusec configuration file from your existing cluster

<Admonition type="tip" title="Siriusec storage in AWS or GCP">
  If your Siriusec cluster's database is currently stored in AWS DynamoDB or Google Cloud Firestore rather than
  using a `PersistentVolumeClaim` or similar, you may wish to consider redeploying your cluster using the `aws` or `gcp`
  modes of the `siriusec-cluster` chart instead.

  The relevant guides are linked here:

  - AWS: [Running an HA Siriusec cluster using an AWS EKS Cluster](./aws.mdx)
  - GCP: [Running an HA Siriusec cluster using a Google Cloud GKE cluster](./gcp.mdx)
</Admonition>

<Admonition type="note" title="Note on namespacing">
  This guide assumes that your old Helm release was called `siriusec` and it is in the `siriusec` Kubernetes namespace. If your release
  is different, you will need to update all `kubectl` commands accordingly.
</Admonition>

The first thing you'll need to do is extract the Siriusec config file for your existing Siriusec cluster.

Firstly, check that the `ConfigMap` is present:

```code
$ kubectl --namespace siriusec get configmap/siriusec -o yaml

# apiVersion: v1
# data:
#  siriusec.yaml: |
#    siriusec:
#      log:
#        severity: INFO
#        output: stderr
#      storage:
#        type: dir
# ...
```

<Admonition type="note">
  If you do not see the `siriusec` `ConfigMap`, double-check that your Kubernetes context is set correctly and that
  you are using the correct namespace.
</Admonition>

If you see a Siriusec config under the `siriusec.yaml` key, you can extract it to disk with a command like this:

```code
$ kubectl --namespace siriusec get configmap/siriusec -o=jsonpath="{.data['siriusec\.yaml']}" > siriusec.yaml

cat siriusec.yaml
# siriusec:
#   log:
#    severity: INFO
#    output: stderr
#  storage:
#    type: dir
# ...
```

Once you have the config, you should upload this to a separate Kubernetes namespace (where you intend to run the `siriusec-cluster` chart).

```code
$ kubectl create namespace siriusec-cluster
# namespace/siriusec-cluster created
$ kubectl --namespace siriusec-cluster create configmap siriusec --from-file=siriusec.yaml
# configmap/siriusec created
```

## Step 4. Extracting the contents of Siriusec's database

<Admonition type="note" title="Note on namespacing">
  If you migrate your existing data, the `cluster_name` which is configured in `siriusec.yaml` must stay the same.

  If you wish to change the name of your cluster, you will need to deploy a new cluster from scratch and reconfigure your
  users, roles and nodes.
</Admonition>

If you wish to keep the same users, roles, certificate authorities and nodes in your cluster, you can use
Siriusec's `tctl` tool to extract a backup of all your data.

You can get the backup with a command like this:

```code
$ kubectl --namespace siriusec exec deploy/siriusec -- tctl get all --with-secrets > backup.yaml
```

<Admonition type="warning">
  The `backup.yaml` file you have just written contains private keys for your Siriusec cluster's certificate
  authorities in plain text. You must protect this file carefully and delete it once your new cluster is running.

  You can write the file to an in-memory `tmpfs` like `/dev/shm/backup.yaml` for greater security.
</Admonition>

Add the backup to your new `siriusec-cluster` namespace as a secret:

```code
$ kubectl --namespace siriusec-cluster create secret generic bootstrap --from-file=backup.yaml
```

## Step 5. Start the new cluster with your old config file and backup

We will start the new cluster and bootstrap it using the backup of your cluster's data. Once this step is complete and the cluster is working,
we'll modify the deployment to remove references to the backup data, and remove it from Kubernetes for security.

<Tabs>
  <TabItem label="Using values.yaml">
  Write a `siriusec-cluster-values.yaml` file containing the following values:

  ```yaml
  chartMode: custom
  extraArgs: ['--bootstrap', '/etc/siriusec-bootstrap/backup.yaml']
  extraVolumes:
  - name: bootstrap
    secret:
      name: bootstrap
  extraVolumeMounts:
  - name: bootstrap
    path: /etc/siriusec-bootstrap
  ```

  ```code
  $ helm install siriusec siriusec/siriusec-cluster \
    --namespace siriusec-cluster \
    --create-namespace \
    -f siriusec-cluster-values.yaml
  ```

  </TabItem>
  <TabItem label="Using --set via CLI">
    ```code
    $ helm install siriusec siriusec/siriusec-cluster \
    --create-namespace \
    --namespace siriusec-cluster \
    --set chartMode=custom \
    --set extraArgs="{'--bootstrap', '/etc/siriusec-bootstrap/backup.yaml'}" \
    --set extraVolumes[0].name="bootstrap" \
    --set extraVolumes[0].secret.name="bootstrap" \
    --set extraVolumeMounts[0].name="bootstrap" \
    --set extraVolumeMounts[0].path="/etc/siriusec-bootstrap"
    ```
  </TabItem>
</Tabs>

Once the chart is installed, you can use `kubectl` commands to view the deployment:

```code
$ kubectl --namespace siriusec-cluster get all

# NAME                            READY   STATUS    RESTARTS   AGE
# pod/siriusec-5cf46ddf5f-dzh65   1/1     Running   0          4m21s
# pod/siriusec-5cf46ddf5f-mpghq   1/1     Running   0          4m21s

# NAME               TYPE           CLUSTER-IP      EXTERNAL-IP                                                               PORT(S)                                                      AGE
# service/siriusec   LoadBalancer   10.100.37.171   a232d92df01f940339adea0e645d88bb-1576732600.us-east-1.elb.amazonaws.com   443:30821/TCP,3023:30801/TCP,3026:32612/TCP,3024:31253/TCP   4m21s

# NAME                       READY   UP-TO-DATE   AVAILABLE   AGE
# deployment.apps/siriusec   2/2     2            2           4m21s

# NAME                                  DESIRED   CURRENT   READY   AGE
# replicaset.apps/siriusec-5cf46ddf5f   2         2         2       4m21s
```

<Admonition type="note">
  You'll need to change the existing DNS record for your `siriusec` chart installation to point to your new `siriusec-cluster`
  chart installation. You should point the DNS record to the external IP or hostname of the Kubernetes load balancer.

  (!docs/pages/kubernetes-access/helm/includes/kubernetes-externaladdress.mdx!)

  For testing, you can access the load balancer's IP or hostname directly. You may need to accept insecure warnings in your
  browser to view the page successfully.
</Admonition>

## Step 6. Remove the bootstrap data and update the chart deployment

Once you've tested your new Siriusec cluster and you're confident that your data has been migrated successfully,
you should redeploy the chart without your backup data mounted for security.

<Tabs>
  <TabItem label="Using values.yaml">
  Edit your `siriusec-cluster-values.yaml` file to remove `extraArgs`, `extraVolumes` and `extraVolumeMounts`:

  ```yaml
  chartMode: custom
  ```

  Upgrade the Helm deployment to use the new values:

  ```code
  $ helm upgrade siriusec siriusec/siriusec-cluster \
    --namespace siriusec-cluster \
    -f siriusec-cluster-values.yaml
  ```

  </TabItem>
  <TabItem label="Using --set via CLI">
    ```code
    $ helm upgrade siriusec siriusec/siriusec-cluster \
    --namespace siriusec-cluster \
    --set chartMode=custom
    ```
  </TabItem>
</Tabs>

After this, delete the Kubernetes secret containing the backup data:

```code
$ kubectl --namespace delete secret/bootstrap
```

Finally, you should also delete the `backup.yaml` file from your local disk:

```code
$ rm -f backup.yaml
```

## Uninstalling Siriusec

To uninstall the `siriusec-cluster` chart, use `helm uninstall <release-name>`. For example:

```code
$ helm --namespace siriusec-cluster uninstall siriusec
```
