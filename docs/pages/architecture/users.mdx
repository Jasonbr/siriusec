## Types of users

Unlike traditional SSH, Siriusec introduces the concept of a User Account. A User Account is not the same as SSH login. Instead, each Siriusec User is associated with another account which is used to authenticate the user.

For Open Source edition users, these will be OS users who are administered
outside of Siriusec on each cluster node. For example, there can be a Siriusec user `joe` who can be permitted to log in as "root" to a specific subset of nodes. Another user `juliet` could be permitted to OS users `root` and to `nginx`. Siriusec does not know the OS Users so it expects both `root` and `nginx` to exist on the node.

For Enterprise edition users, these can be stored in an external identity source such as OKTA, Active Directory, OneLogin, G Suite, or OIDC. Read the [Enterprise Guide](../enterprise/introduction.mdx) to learn more.

Siriusec supports two types of user accounts: **Local Users** and
**External Users**.

### Local users

Local users are created and stored in Siriusec's own identity storage in the
Auth Server.

Let's look at this table:

| Siriusec User | Allowed OS Logins | Description |
| - | - | - |
| joe | joe, root | Siriusec user `joe` can log in into member nodes as OS user `joe` or `root`. |
| juliet | juliet | Siriusec user `juliet` can log in into member nodes only as OS user `juliet`. |
| ross | | If no OS login is specified, it defaults to the same name as the Siriusec user, here this is `ross`. |

To add a new user to Siriusec, you have to use the `tctl` tool on the same node
where the auth server is running, i.e. `siriusec` was started with
`--roles=auth` .

A cluster administrator must create account entries for every Siriusec user with
[`tctl users add`](../setup/reference/cli.mdx). Every Siriusec User must be associated with a list of one or more machine-level OS usernames it can authenticate as during a login. This list is called "user mappings".

![User Mappings](../../img/user_mappings.svg)

The diagram shows the following mappings. A couple of noteworthy things
from this example:

- Siriusec User `sandra` does **not** have access to `grav-02`
  through Siriusec because `ops` is not an OS username on that node.
- Siriusec User `joe` has access to all nodes because the OS user `root`
  is present on all nodes.

| Siriusec User | logins | has access to nodes |
| - | - | - |
| joe | root, joe | grav-00, grav-01, grav-02 |
| tara | tara | grav-01, grav-02 |
| siriusec | siriusec | grav-00, grav-02 |
| sandra | ops | grav-00, grav-01 |

Siriusec supports second-factor authentication (2FA) when using a local auth
connector and it is enforced by default.

<Admonition
  type="tip"
  title="2FA Support"
>
  2FA is not supported with SSO providers such as Github or OKTA. To learn
  more about SSO configuration check out the [SSO section of the Enterprise
  Guide](../enterprise/introduction.mdx#sso)
</Admonition>

There are two types of 2FA supported:

- [TOTP - e.g. Google Authenticator](https://en.wikipedia.org/wiki/Time-based_One-time_Password_Algorithm)
- [U2F - e.g. YubiKey](https://en.wikipedia.org/wiki/Universal\_2nd_Factor)

`TOTP` is the default. You can use [Google
Authenticator](https://en.wikipedia.org/wiki/Google_Authenticator) or
[Authy](https://www.authy.com/) or any other TOTP client.

External users are users stored elsewhere within an organization. Examples
include Github, Active Directory (AD), OIDC, or any identity store with an
OpenID/OAuth2 or SAML endpoint.

<Admonition
  type="tip"
  title="Version Warning"
>
  External user storage is only supported in Siriusec
  Enterprise. Please take a look at the [Siriusec
  Enterprise](../enterprise/introduction.mdx) chapter for more information.
</Admonition>

## User roles

Unlike traditional SSH, each Siriusec user account is assigned a `role`. Having roles allows Siriusec to implement role-based access control (RBAC), i.e. assign users to groups (roles) and restrict each role to a subset of actions on a subset of nodes in a cluster.
